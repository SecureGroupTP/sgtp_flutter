# Circle Video Messages — Implementation Pipeline

## Обзор

Кружок (circle video message) — короткое видео, записанное фронтальной камерой, отображаемое в форме круга. Финальный артефакт — квадратное видео NxN пикселей, где полезная область вписана в круг, а маска круга накладывается на стороне отображения (UI).

**Ключевой принцип:** видеофайл всегда хранится и передаётся как обычный квадратный MP4. Круглая форма — это исключительно UI-маска при записи и воспроизведении. Это гарантирует совместимость с любым плеером/транспортом.

---

## 1. Архитектура: слои системы

```
┌─────────────────────────────────────────────────────┐
│  UI Layer          — превью, маска круга, таймер    │
├─────────────────────────────────────────────────────┤
│  Capture Layer     — камера, микрофон, ориентация   │
├─────────────────────────────────────────────────────┤
│  Processing Layer  — crop, mirror, orientation fix  │
├─────────────────────────────────────────────────────┤
│  Encoding Layer    — кодирование в финальный MP4    │
├─────────────────────────────────────────────────────┤
│  Transport Layer   — загрузка, метаданные           │
├─────────────────────────────────────────────────────┤
│  Playback Layer    — декодирование, маска, звук     │
└─────────────────────────────────────────────────────┘
```

## 2. Capture Layer — захват с камеры

### 2.1. Выбор камеры

```
При инициализации:
  1. Запросить список доступных камер
  2. Найти фронтальную камеру (facing: front / user)
  3. Если фронталки нет → fallback на заднюю камеру
  4. Если камер нет вообще → показать ошибку, не крашить
  5. Сохранить ID текущей камеры в состояние
```

**Баг-превенция — отсутствие фронталки:**
- Десктопные устройства, эмуляторы, старые планшеты могут не иметь фронтальной камеры.
- Никогда не хардкодить `facing: front`. Всегда проверять наличие и делать fallback.
- При fallback на заднюю камеру: отключить зеркалирование (см. раздел 3).

### 2.2. Параметры захвата

```
Целевое разрешение захвата:
  - Идеальное:  480×480  (баланс качества и размера)
  - Минимальное: 240×240 (для слабых устройств)
  - Максимальное: 720×720 (нет смысла больше для кружка)

Если камера не поддерживает квадратное разрешение:
  → Запросить ближайшее разрешение с aspect ratio ≥ 1:1
  → Предпочтительно 4:3 (480×640, 720×960)
  → Crop до квадрата выполняется в Processing Layer
  → НИКОГДА не запрашивать 16:9 если есть 4:3

FPS: 30 (достаточно для кружка, экономит размер)
Максимальная длительность: 60 секунд (хардкод-лимит)
```

### 2.3. Захват аудио

```
Параметры микрофона:
  - Моно (1 канал) — стерео бессмысленно для кружка
  - Sample rate: 44100 Hz (или 48000 если платформа требует)
  - Кодек: AAC (универсальная совместимость)

Обработка:
  - Проверить разрешение на микрофон ДО начала записи
  - Если микрофон недоступен → записывать без звука, 
    пометить флагом has_audio: false
  - Синхронизация A/V: использовать единый таймстемп-источник
    для видео и аудио фреймов
```

### 2.4. Свап камеры во время записи

```
Алгоритм свапа камеры:
  1. НЕ останавливать запись/энкодер
  2. НЕ останавливать захват аудио (аудио пишется непрерывно)
  3. Открыть новый источник камеры (front ↔ back)
  4. Дождаться первого фрейма с нового источника
  5. Атомарно переключить видеопоток на новый источник
  6. Закрыть старый источник камеры
  7. Обновить флаг is_front_camera → пересчитать зеркалирование

Критические моменты:
  - Между шагами 3 и 4 будет задержка (~100-500ms)
  - В это время показывать последний фрейм (freeze), 
    НЕ чёрный экран
  - В энкодер в это время можно дублировать последний фрейм
    или пропустить фреймы (encoder сам обработает gap)
  - Аудио НЕ прерывается ни при каких обстоятельствах
```

## 3. Processing Layer — обработка фреймов

Это самый критичный слой, где возникает большинство багов.

### 3.1. Crop до квадрата

```
Входной фрейм: W × H (может быть любым соотношением)
Целевой фрейм: S × S, где S = min(W, H)

Вычисление области обрезки:
  crop_x = (W - S) / 2    ← центрирование по горизонтали
  crop_y = (H - S) / 2    ← центрирование по вертикали
  crop_rect = (crop_x, crop_y, S, S)

Применение:
  → Обрезать каждый фрейм ДО передачи в энкодер
  → Или настроить энкодер на input crop (эффективнее)
```

**Когда кропать — два подхода:**

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| Crop в реальном времени (при записи) | Энкодер получает готовые фреймы, проще отладка | Нагрузка на CPU во время записи |
| Crop после записи (пост-процессинг) | Запись быстрее, можно менять кроп-область | Пауза перед отправкой, нужен полный прогон файла |

**Рекомендация:** crop при записи. Это проще, надёжнее и устраняет целый класс багов с метаданными ориентации.

### 3.2. Зеркалирование (Mirror)

Это главный источник бага "инвертированное видео".

```
Правила зеркалирования:

ФРОНТАЛЬНАЯ камера:
  ┌──────────────────────────────────────────────────────┐
  │ Превью на экране: ЗЕРКАЛИТЬ (horizontal flip)       │
  │   → Пользователь видит себя как в зеркале           │
  │   → Это ожидаемое поведение, люди привыкли           │
  │                                                      │
  │ Записанное видео: ЗЕРКАЛИТЬ (horizontal flip)        │
  │   → Записывается как в превью                        │
  │   → Текст будет зеркальным — это ОК для кружков     │
  │   → Telegram/WhatsApp так делают                     │
  │                                                      │
  │ Альтернатива: НЕ зеркалить запись                    │
  │   → Тогда текст нормальный, но жесты инвертированы  │
  │   → Выбор за продуктом, но ДОЛЖЕН быть консистентным│
  └──────────────────────────────────────────────────────┘

ЗАДНЯЯ камера:
  ┌──────────────────────────────────────────────────────┐
  │ Превью: НЕ зеркалить                                │
  │ Запись: НЕ зеркалить                                 │
  └──────────────────────────────────────────────────────┘

СВАП камеры:
  → При каждом свапе пересчитывать флаг зеркалирования
  → Фреймы после свапа обрабатываются по новому правилу
```

**Как именно зеркалить:**
```
Horizontal flip = отразить по оси Y
Для каждого пикселя: new_x = (width - 1) - old_x

Или через матрицу трансформации:
  scale(-1, 1)  +  translate(width, 0)

На уровне GPU/шейдера:
  texture coordinate: u = 1.0 - u
```

### 3.3. Ориентация (Orientation) — КРИТИЧЕСКИЙ РАЗДЕЛ

Это самый частый источник багов. Видео переворачивается, ложится набок, отображается вверх ногами.

```
ПРОБЛЕМА:
  Камера телефона имеет фиксированную физическую ориентацию сенсора.
  Сенсор всегда пишет в landscape.
  Для корректного отображения используется rotation metadata
  в контейнере (MP4 → moov/trak/tkhd → rotation matrix).

  НО: многие плееры/транспорты ТЕРЯЮТ или ИГНОРИРУЮТ эту метаданные.

РЕШЕНИЕ — "BAKE ROTATION":
  Применить поворот к пикселям ФИЗИЧЕСКИ, а не через метаданные.
  Записать rotation = 0 в метаданных.
  Тогда видео корректно в ЛЮБОМ плеере.
```

```
Алгоритм bake rotation:

1. Получить ориентацию устройства (0°, 90°, 180°, 270°)
   Источник: акселерометр / датчик ориентации / API устройства

2. Получить "native" ориентацию сенсора камеры
   (обычно 90° для фронталки на Android, 0° на iOS)

3. Вычислить rotation_needed:
   
   На Android:
     Для задней камеры:
       rotation = (sensor_orientation - device_orientation + 360) % 360
     Для фронтальной камеры:
       rotation = (sensor_orientation + device_orientation) % 360
       (+ flip, см. зеркалирование)
   
   На iOS:
     AVCaptureConnection.videoOrientation задаёт ориентацию.
     Установить её = текущей ориентации устройства.
     iOS bake-ит поворот автоматически если правильно 
     настроить connection.

   На Web (MediaStream):
     Браузер обычно отдаёт фреймы уже повёрнутые.
     НО: проверить через MediaStreamTrack.getSettings().facingMode
     и ImageCapture API.

4. Применить поворот к каждому фрейму ПЕРЕД энкодированием:
   - 0°:   ничего не делать
   - 90°:  transpose + flip horizontal
   - 180°: flip both axes
   - 270°: transpose + flip vertical

5. В metadata записываемого файла: rotation = 0
```

**Чеклист ориентации — проверь все кейсы:**
```
□ Телефон вертикально (portrait) — фронтальная камера
□ Телефон вертикально (portrait) — задняя камера
□ Телефон горизонтально (landscape left) — обе камеры
□ Телефон горизонтально (landscape right) — обе камеры
□ Телефон вверх ногами (portrait inverted) — обе камеры
□ Свап камеры во время записи при любой ориентации
□ Планшет (другие sensor orientations)
□ Десктоп (webcam, rotation обычно 0°)
```

### 3.4. Scale / Resize

```
После crop и rotation, фрейм = S × S.
Если S ≠ target_size:
  Resize до target_size × target_size
  Алгоритм: bilinear (быстро) или lanczos (качественнее)

target_size выбирается один раз при старте записи 
и НЕ меняется до конца.
```

### 3.5. Порядок операций — КРИТИЧЕСКИ ВАЖЕН

```
Правильный порядок обработки каждого фрейма:

  raw frame from camera
       │
       ▼
  ① ROTATE (bake orientation)
       │
       ▼
  ② CROP (вырезать квадрат из центра)
       │
       ▼
  ③ MIRROR (flip по горизонтали если фронталка)
       │
       ▼
  ④ RESIZE (до целевого размера NxN)
       │
       ▼
  frame ready for encoder

⚠ Если поменять порядок — будут баги:
  - CROP до ROTATE → обрежется не та область
  - MIRROR до ROTATE → зеркалирование по неправильной оси
  - RESIZE до CROP → потеря качества
```

## 4. Encoding Layer — кодирование

### 4.1. Параметры энкодера

```
Контейнер:  MP4 (moov atom в начале файла — для стриминга)
Видео кодек: H.264 Baseline Profile
  - Baseline: максимальная совместимость
  - НЕ использовать High Profile для кружков
  - Pixel format: yuv420p (обязательно, иначе многие 
    плееры не откроют)

Видео параметры:
  - Resolution: 480×480 (или выбранный target_size)
  - FPS: 30
  - Bitrate: 800 kbps - 1200 kbps (variable bitrate)
  - Keyframe interval: 1-2 секунды (каждые 30-60 фреймов)
  - Rotation metadata: 0 (мы уже bake-нули)

Аудио кодек: AAC-LC
  - Sample rate: 44100 Hz
  - Channels: 1 (mono)
  - Bitrate: 64 kbps (достаточно для голоса)
```

### 4.2. Финализация файла

```
После остановки записи:
  1. Flush все буферы энкодера
  2. Записать оставшиеся фреймы
  3. Финализировать контейнер (закрыть moov atom)
  4. ОБЯЗАТЕЛЬНО: переместить moov atom в начало файла
     (faststart / qt-faststart)
     → Без этого видео не будет стримиться,
       придётся скачивать полностью перед воспроизведением

Валидация финального файла:
  □ Разрешение ровно N×N (квадрат)
  □ Rotation в metadata = 0
  □ Pixel format = yuv420p
  □ Длительность ≤ 60 секунд
  □ Размер файла разумный (< 15 MB для 60 секунд)
  □ Файл играется в стороннем плеере корректно (не повёрнут)
  □ Аудио синхронизировано с видео
```

## 5. UI Layer — отображение при записи

### 5.1. Превью камеры

```
Структура UI при записи:

  ┌─────────────────────────────┐
  │                             │
  │    ┌───────────────┐        │
  │    │   ╭───────╮   │        │
  │    │   │       │   │        │  ← Квадратное превью камеры
  │    │   │ (face)│   │        │     с круглой маской поверх
  │    │   │       │   │        │
  │    │   ╰───────╯   │        │
  │    └───────────────┘        │
  │                             │
  │       ◉ 0:42  ⟲             │  ← Кнопка записи, таймер, 
  │                             │     свап камеры
  └─────────────────────────────┘

Круглая маска:
  - CSS: border-radius: 50%; overflow: hidden;
  - Или через Canvas/GL: circular clipping path
  - Маска ТОЛЬКО визуальная, в энкодер идёт полный квадрат

Превью ДОЛЖНО показывать:
  - Зеркалированное изображение для фронталки
  - НЕ зеркалированное для задней камеры
  - Уже обрезанное до квадрата (чтобы WYSIWYG)
```

### 5.2. Индикация записи

```
Обязательные элементы:
  - Красная точка / пульсация — идёт запись
  - Таймер — текущая длительность (MM:SS)
  - Прогресс-бар по кругу (кольцо вокруг превью)
    заполняется от 0 до max_duration
  - Кнопка свапа камеры
  - Кнопка остановки записи

При достижении max_duration:
  → Автоматическая остановка записи
  → Плавная анимация завершения
```

## 6. Transport Layer — отправка

### 6.1. Метаданные сообщения

```json
{
  "type": "circle_video",
  "file_id": "...",
  "duration_seconds": 42,
  "dimensions": { "width": 480, "height": 480 },
  "has_audio": true,
  "file_size_bytes": 2340000,
  "thumbnail": "<base64 первого кадра, JPEG, 120×120>"
}
```

### 6.2. Thumbnail — генерация

```
Генерация thumbnail:
  1. Взять первый (или ключевой) фрейм
  2. Resize до 120×120 или меньше
  3. Кодировать как JPEG, quality 60-70
  4. Отправить inline (base64) или как отдельный файл
  5. Thumbnail тоже квадратный, маска круга на стороне UI

Назначение:
  - Показывать пока видео загружается
  - Показывать в списке чатов
  - Показывать при плохом интернете
```

### 6.3. Thumbnail — защита от чёрного/битого превью

Чёрный или отсутствующий thumbnail — одна из самых заметных
проблем для пользователя: кружок выглядит сломанным.

```
ПРОБЛЕМА — ЧЁРНЫЙ ПЕРВЫЙ КАДР:
  Первый кадр видео с камеры часто бывает:
  - Полностью чёрным (камера ещё не отдала экспозицию)
  - Засвеченным (автоэкспозиция не успела сработать)
  - Размытым (автофокус не сработал)
  - Зелёным / артефактным (кодек не стабилизировался)

  Это особенно вероятно на Android, где первые 2-5 фреймов
  с камеры часто мусорные.

РЕШЕНИЕ — УМНЫЙ ВЫБОР КАДРА ДЛЯ THUMBNAIL:

  Алгоритм:
  1. НЕ брать первый кадр слепо
  2. Взять кадр с позиции ~0.5с от начала записи
     (или 15-й фрейм при 30fps)
  3. Проверить кадр на "чёрность":
     
     is_black_frame(frame):
       average_brightness = mean(frame.pixels.luminance)
       return average_brightness < 15  
       // порог 15 из 255, подобрать эмпирически

  4. Если кадр чёрный → сдвинуться вперёд на 0.5с, 
     повторить проверку
  5. Максимум 5 попыток, шаг 0.5с
  6. Если все кадры чёрные → взять кадр с середины видео
  7. Если и середина чёрная → сгенерировать placeholder

  Аналогично проверять на "засвеченность":
     is_overexposed(frame):
       average_brightness = mean(frame.pixels.luminance)
       return average_brightness > 245

  Финальный fallback chain:
    кадр@0.5с → кадр@1.0с → кадр@1.5с → кадр@2.0с 
    → кадр@2.5с → кадр@mid → placeholder
```

```
ПРОБЛЕМА — THUMBNAIL НЕ ЗАГРУЖАЕТСЯ НА СТОРОНЕ ПОЛУЧАТЕЛЯ:

  Причины:
  - Сеть оборвалась после получения метаданных но до 
    загрузки thumbnail
  - CDN потерял thumbnail-файл (если он отдельный)
  - Base64 thumbnail слишком большой и был обрезан 
    транспортом
  - Формат thumbnail не поддерживается клиентом

РЕШЕНИЕ — МНОГОУРОВНЕВЫЙ FALLBACK НА СТОРОНЕ ПОЛУЧАТЕЛЯ:

  Приоритет отображения (сверху вниз):

  ┌─ 1. Inline thumbnail из метаданных сообщения
  │     (base64 JPEG, приходит вместе с сообщением)
  │
  ├─ 2. Отдельный thumbnail-файл с сервера
  │     (если inline не пришёл / битый)
  │
  ├─ 3. Извлечь первый кадр из самого видеофайла
  │     (если видео уже скачано или частично буферизовано)
  │     Метод: seek(0) → grab frame → display
  │
  ├─ 4. Средний solid-color placeholder
  │     (серый круг с иконкой ▶ play)
  │     Цвет: средний цвет из palette метаданных,
  │     или нейтральный серый #808080
  │
  └─ 5. Skeleton / shimmer анимация
        (пока не загрузится хоть что-то)

  Переход между уровнями:
  - Начинать с уровня 5 (skeleton) сразу при получении 
    сообщения
  - Параллельно пытаться загрузить уровни 1-3
  - Как только любой уровень готов → показать его
  - При получении более качественного уровня → заменить
    плавно (crossfade 150ms, без мигания)
```

```
ПРОБЛЕМА — THUMBNAIL НЕ СОВПАДАЕТ С ВИДЕО:

  Причины:
  - Thumbnail сгенерирован до применения mirror/rotation
  - Thumbnail сгенерирован из raw-фрейма, а видео 
    перекодировано с другими параметрами
  - При перекодировании на сервере видео изменилось, 
    а thumbnail остался старым

РЕШЕНИЕ:
  ✓ Генерировать thumbnail ИЗ ФИНАЛЬНОГО файла, 
    а не из raw-потока камеры
  ✓ Последовательность: запись → encode → finalize MP4 
    → extract frame from MP4 → make thumbnail
  ✓ Это гарантирует что thumbnail = то что увидит получатель
  ✓ Если сервер перекодирует видео, сервер должен 
    перегенерировать thumbnail тоже

  Если thumbnail генерируется из raw-потока (для скорости):
  ✓ Применить тот же pipeline что и к видео:
    rotate → crop → mirror → resize
  ✓ Но это дублирование логики и источник рассинхрона,
    поэтому предпочтительнее извлекать из готового MP4
```

```
ПРОБЛЕМА — МЕРЦАНИЕ / ПЕРЕРИСОВКА THUMBNAIL:

  Причины:
  - Thumbnail загружается, показывается, потом видео
    начинает грузиться → первый кадр видео отличается 
    от thumbnail → визуальный "прыжок"
  - Thumbnail показывается, потом исчезает на мгновение
    когда видеоплеер инициализируется

РЕШЕНИЕ:
  ✓ Thumbnail-слой ПОВЕРХ video-элемента (z-index выше)
  ✓ Скрывать thumbnail только ПОСЛЕ того как видеоплеер
    отрендерил первый кадр и готов к воспроизведению
  ✓ Событие для скрытия: onFirstFrameRendered / 
    canplay + requestAnimationFrame
  ✓ Transition: opacity 150ms (не display:none мгновенно)

  Layering:
    ┌──────────────┐
    │  ● circle    │ ← CSS clip-path / border-radius
    │  ┌────────┐  │
    │  │ thumb  │  │ ← z-index: 2, opacity transition
    │  │ ┌────┐ │  │
    │  │ │vid │ │  │ ← z-index: 1, рендерится под thumb
    │  │ └────┘ │  │
    │  └────────┘  │
    └──────────────┘
```

## 7. Playback Layer — воспроизведение

### 7.1. Отображение в чате

```
Получатель отображает кружок:

  ┌──────────────────────────────────┐
  │  Alice                           │
  │    ╭───────╮                     │
  │    │  ▶    │  0:42               │
  │    │       │                     │
  │    ╰───────╯                     │
  │                           14:30  │
  └──────────────────────────────────┘

Правила:
  - Видео отображается с круглой CSS/canvas маской
  - Первый кадр (thumbnail) виден до нажатия
  - По нажатию: воспроизведение (автоплей опционален)
  - Звук по умолчанию выключен, включается по нажатию 
    или по настройке
  - Прогресс отображается кольцом вокруг кружка
  - После окончания: возврат к первому кадру или loop
```

### 7.2. Защита от сжатия и искажений

```
ПРОБЛЕМА: Некоторые устройства/прокси/CDN могут 
ресайзить или перекодировать видео, нарушив квадрат.

ЗАЩИТА:
  1. Проверять dimensions из метаданных при получении
  2. Если реальное разрешение ≠ метаданным → перекропить
  3. Если aspect ratio ≠ 1:1 → пересчитать с padding/crop:
     
     if width > height:
       crop_x = (width - height) / 2
       display_rect = (crop_x, 0, height, height)
     else:
       crop_y = (height - width) / 2
       display_rect = (0, crop_y, width, width)
  
  4. НИКОГДА не растягивать (stretch) — только crop или 
     letterbox
  5. Применить маску круга ПОСЛЕ коррекции aspect ratio
```

### 7.3. Обработка ориентации при воспроизведении

```
Даже если мы bake-нули ориентацию при записи,
на стороне приёма нужна защита:

  1. Прочитать rotation metadata из MP4
  2. Если rotation ≠ 0:
     → Применить обратный поворот при воспроизведении
     → Или перекодировать файл локально с bake
  3. Проверить: первый кадр корректный? 
     Сравнить с thumbnail из метаданных (если есть)
```

## 8. Типовые баги и их решения

### 8.1. Видео перевёрнуто (upside down)

```
Причина: rotation не bake-нут, а плеер получателя 
         игнорирует metadata.

Решение: 
  ✓ Всегда bake rotation в пиксели (раздел 3.3)
  ✓ Писать rotation=0 в metadata
  ✓ Тестировать на 3+ разных плеерах
```

### 8.2. Видео зеркально

```
Причина: фронталка по умолчанию отдаёт не-зеркальное 
         изображение, а flip применялся только к превью.

Решение:
  ✓ Mirror применять к фреймам ДО энкодера, не только 
    к превью
  ✓ Проверить: то что видит пользователь при записи =
    то что видит получатель
```

### 8.3. Видео сжато / не квадратное

```
Причина: транспорт или CDN перекодировал файл и 
         изменил разрешение.

Решение:
  ✓ Передавать dimensions в метаданных сообщения
  ✓ На стороне получателя: коррекция aspect ratio (7.2)
  ✓ Если возможно: запретить перекодирование на стороне
    сервера, или перекодировать с сохранением aspect ratio
```

### 8.4. Видео повёрнуто на 90°

```
Причина: сенсор камеры в landscape, rotation не применён.

Решение:
  ✓ Раздел 3.3 — bake rotation
  ✓ Особое внимание: Android, где sensor_orientation 
    зависит от производителя
```

### 8.5. Аудио рассинхронизировано

```
Причина: аудио и видео имеют разные timestamp-источники,
         или при свапе камеры возникает gap в видео 
         но не в аудио.

Решение:
  ✓ Единый clock source для A/V timestamps
  ✓ При свапе камеры: дублировать фреймы чтобы закрыть gap
  ✓ Валидация после записи: 
    |audio_duration - video_duration| < 100ms
```

### 8.6. Чёрные фреймы при свапе камеры

```
Причина: новая камера ещё не отдаёт фреймы, а старая 
         уже закрыта.

Решение:
  ✓ Не закрывать старый источник до получения первого 
    фрейма с нового
  ✓ Freeze на последнем фрейме в превью
  ✓ В энкодер: дублировать последний фрейм или дропнуть
    gap (плеер интерполирует)
```

### 8.7. Камера недоступна

```
Причина: нет разрешения, камера занята другим приложением,
         или физически отсутствует.

Решение:
  ✓ Graceful degradation на каждом этапе
  ✓ Нет фронталки → задняя камера (без mirror)
  ✓ Нет камер → сообщение об ошибке
  ✓ Камера занята → retry с задержкой (2-3 попытки), 
    потом ошибка
  ✓ Разрешение отозвано во время записи → остановить 
    запись, сохранить что есть
```

### 8.8. Чёрный / пустой thumbnail

```
Причина: первый кадр с камеры чёрный (экспозиция не готова),
         или thumbnail сгенерирован до финализации видео.

Решение:
  ✓ Не использовать первый кадр — брать кадр с ~0.5с
  ✓ Проверять brightness кадра перед использованием
  ✓ Fallback chain: кадр@0.5с → @1.0с → @mid → placeholder
  ✓ Подробный алгоритм — раздел 6.3
```

### 8.9. Thumbnail не загружается у получателя

```
Причина: сеть оборвалась, CDN потерял файл, base64 обрезан
         транспортом, или клиент не поддерживает формат.

Решение:
  ✓ Многоуровневый fallback (раздел 6.3):
    inline base64 → CDN URL → extract from video → placeholder
  ✓ Начинать с skeleton-анимации, заменять по мере загрузки
  ✓ Dominant color из метаданных как фон placeholder
  ✓ Никогда не показывать пустое место / broken image
```

### 8.10. Thumbnail не совпадает с видео

```
Причина: thumbnail сгенерирован из raw-потока камеры 
         без mirror/rotation, а видео прошло полный pipeline.
         Или сервер перекодировал видео но не пересоздал 
         thumbnail.

Решение:
  ✓ Генерировать thumbnail ИЗ ГОТОВОГО MP4, не из raw-потока
  ✓ Если генерируется из raw: применить тот же pipeline
    (rotate → crop → mirror → resize)
  ✓ Если сервер перекодирует видео → перегенерить thumbnail
```

### 8.11. Мерцание при переходе от thumbnail к видео

```
Причина: thumbnail исчезает мгновенно когда видеоплеер 
         инициализируется, но первый кадр ещё не отрендерен.
         Результат: thumbnail → чёрное мигание → видео.

Решение:
  ✓ Thumbnail-слой ПОВЕРХ video-элемента (z-index)
  ✓ Скрывать thumbnail ТОЛЬКО после рендера первого 
    кадра видео (canplay + requestAnimationFrame)
  ✓ Скрывать через opacity transition (150ms), 
    не через display:none
  ✓ Подробная схема layering — раздел 6.3
```

## 9. Чеклист для тестирования

```
ЗАПИСЬ:
  □ Фронтальная камера — portrait
  □ Фронтальная камера — landscape (оба направления)
  □ Задняя камера — portrait
  □ Задняя камера — landscape (оба направления)
  □ Свап камеры посередине записи
  □ Свап камеры несколько раз
  □ Запись ровно 60 секунд (автоостановка)
  □ Запись 1 секунду (минимум)
  □ Запись без доступа к микрофону
  □ Запись на слабом устройстве (проверить FPS drop)

ФАЙЛ:
  □ Открыть в VLC — не перевёрнуто, квадрат, звук есть
  □ Открыть в QuickTime — аналогично
  □ Открыть в браузере <video> — аналогично  
  □ Проверить metadata: rotation=0, dimensions NxN
  □ Размер файла адекватный

ОТОБРАЖЕНИЕ:
  □ Круглая маска корректна на разных размерах экрана
  □ Thumbnail отображается до воспроизведения
  □ Прогресс-бар корректен
  □ Повторное воспроизведение работает
  □ Кружок от iOS показывается на Android и наоборот
  □ Кружок корректен после пересылки (forward)
  □ Кружок корректен после скачивания и повторной загрузки
```
